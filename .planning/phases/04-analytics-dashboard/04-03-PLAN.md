<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js page (server-rendered + client-hydrated) | The `Trade.strategy_name` value originates from the API (already authenticated; already SQLAlchemy-parameterized — see Wave 1 threats T-04-01..T-04-03) and reaches the page through state. |
| User → DOM via `<a href>` | The strategy name is interpolated into a URL via `encodeURIComponent` and rendered as text inside `<a>{t.strategy_name}</a>` — no `dangerouslySetInnerHTML`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-W3-01 | XSS via strategy_name in URL | Tampering | mitigate | The strategy name is wrapped in `encodeURIComponent(t.strategy_name)` before insertion into the `href` template literal. The visible text uses `{t.strategy_name}` as a JSX child — React escapes by default. No `dangerouslySetInnerHTML`. Verified by `grep -nE 'dangerouslySetInnerHTML' dashboard/app/page.tsx` count being unchanged from before this plan to after. |
| T-04-W3-02 | XSS via strategy_name in `<a>{t.strategy_name}</a>` | Tampering | mitigate | React JSX child escaping covers this; HTML special characters in the strategy name (`<`, `>`, `&`, `"`) are auto-escaped to entities. Strategy names today are ASCII-only YAML keys (validated by Pydantic `extra="forbid"` in Phase 02), but defense-in-depth via React's default escaping is preserved. |
| T-04-W3-03 | Information Disclosure via header link | mitigate | The Analytics link is rendered unconditionally on the main dashboard, but the `/analytics` route itself enforces `checkAuth()` (Wave 2 plan). An unauthenticated user who somehow rendered this page (impossible in practice — the main dashboard is also behind `checkAuth`) and clicked the link would be redirected back to `/`. |

No HIGH severity threats. Both edits are presentational; the strategy name was already present in API responses for legitimate authenticated users. No new authentication or authorization surface is introduced.
</threat_model>
