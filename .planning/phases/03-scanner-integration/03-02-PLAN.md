<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API lifespan startup ↔ on-disk SQLite predictions.db | `init_db()` runs migrations; rename touches the on-disk schema once on the first post-deploy boot. |
| API lifespan startup ↔ S3 backup | `_download_db()` pulls latest backup BEFORE init_db runs; the migration applies to whatever schema the backup contains. |
| Scanner async loops ↔ engine | After init_db returns, scanner loops open new sessions; the renamed table is invisible to ORM (StretchOpportunity is gone). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-05 | Tampering / data loss | RENAME executes against an unbacked-up DB and rollback is impossible | mitigate | D-03 pre-deploy gate (S3 backup roundtrip): documented in DEPLOY checklist + manual-only verification row in 03-VALIDATION.md. Idempotent guard via `inspector.get_table_names()` prevents double-rename. RESEARCH Pitfall 4: init_db runs at API lifespan startup BEFORE scanner spawns — no concurrent connections to invalidate prepared statements. |
| T-3-07 | Tampering | SQL injection via `text()` raw queries in _migrate_add_columns | accept | All ALTER strings are static literals — no user input flows into the migration. SQLAlchemy `text()` parameterization not needed for static DDL. |
| T-3-08 | Denial of service | Migration takes too long on large prod DB; SQLite locks while ALTER runs | accept | ALTER TABLE RENAME on SQLite 3.25+ is O(1) (metadata-only update; no row rewrite). ADD COLUMN is also O(1). CREATE INDEX scans the table once — for the trades table (~10K rows in production) this is sub-second. The new `timeout=5` (D-02) handles any incidental BUSY during boot. |
| T-3-09 | Information disclosure | The legacy stretch_opportunities table physically remains as `stretch_opportunities_archived` — old WHAT_IF data persists | accept | INTENTIONAL per D-03: "rename is intended to be permanent for archival access." No new exposure vector — the table is no longer mapped to an ORM and no endpoint reads from it. Phase 4 may add audit-only SELECT access if needed. |
| T-3-10 | Tampering / repudiation | Idempotency guard fails (e.g., partial rename due to power loss) leaving an inconsistent intermediate state | accept | SQLite `ALTER TABLE ... RENAME TO` is atomic at the file system level — completes inside a single COMMIT. If the WAL crashes mid-COMMIT, SQLite's recovery rolls back to pre-rename state on next open. Verified by RESEARCH SQLite caveats. |
</threat_model>
