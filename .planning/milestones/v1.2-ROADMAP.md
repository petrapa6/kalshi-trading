## Overview

Replaced the hardcoded `WHAT_IF_STRATEGIES` shadow-backtest list with a file-driven strategy engine. Named strategies in `strategies.yaml` (multi-trigger OR-of-AND conditions) drive both the backtest simulator and the live scanner via a single `load_strategies()` source of truth. The live scanner fires dry-run trades — Kalshi-price-recorded but no API call — and a new `/analytics` dashboard page surfaces per-strategy P&L, win rate, and a 5-minute-refreshing trade log.

The stretch system (`WHAT_IF_STRATEGIES`, `_evaluate_what_if_strategies`, `stretch_opportunities` ORM) was decommissioned. The `stretch_opportunities` table itself was renamed (not dropped) per a deliberate rollback-safety decision.

## Milestone Summary

**Key Decisions (during v1.2):**

- **STR-04 deviation (RENAME, not DROP):** Production has 6+ months of stretch data; tested rename against S3 backup. Drop = data loss; rename = idempotent + reversible. ORM class deleted (D-20) so fresh DBs don't recreate it.
- **D-13 hardcoded dry_run / yes_price:** Strategies are local-simulation-only by design. Honoring process-level `DRY_RUN` would let real trades fire by accident if the env var is ever flipped. Single boundary mirrors the `extract_cents` invariant.
- **D-16 composite settlement filter:** Real trades always have `strategy_name=NULL`; strategy fires always have it set. The composite filter `dry_run==False OR (dry_run==True AND strategy_name IS NOT NULL)` lets one settlement loop reconcile both populations without branching.
- **D-02 SQLAlchemy `connect_args timeout=5`:** Phase 4 analytics polling competes with scanner per-loop writes against the same SQLite file. Pre-emptive 5-second buffer; revisit if WAL or read replica is needed under load.
- **D-08 sport-family literals (UK terminology):** `football` not `soccer` for trigger fields. ESPN's `sport_path` already uses `soccer/`, so the family literal is the natural terminology boundary. `SPORT_FAMILY_TO_PATHS` is the only translation surface.
- **D-19 `/api/sport-stats` rerouted:** "Played" semantics shift from near-miss rows in `stretch_opportunities` to distinct events scanned in `opportunities`. Phase 4 analytics consumed this convention.
- **CR-01 settled_at writer fix (Phase 04 gap closure):** Both settlement paths (REST `check_settlements` and WS `on_lifecycle`) now write `settled_at = datetime.now(timezone.utc)`; idempotent backfill in `_migrate_add_columns` handles historical rows.

**Key Issues Hit:**

- **CR-01 settled_at writer missing:** Phase 04 verification surfaced that `Trade.settled_at` was never written by Phase 03's settlement code (despite the column existing). Backfilled in 3 atomic commits during the verification re-run; the chart pipeline went from `gaps_found` (6/8) to `complete` (8/8).
- **Wave-0 lint debt understated:** Phase 02 deferred-items.md baseline claimed 16 E501s in scanner.py; actual repo-wide was 4 + 2 + 1 + 9 across multiple files. Cleared in chore commit `613f7e4` to make Criterion #4 pass at full-repo level.
- **D-02 backtest sport override:** Original Phase 02 plan had `trigger.sport` as ESPN sport_path; revised mid-phase (override at 02-CONTEXT.md `Revision — 2026-04-30`) to a sport-family literal (`football`, `basketball`, …). Phase 3 scanner port had to read the addendum, NOT original D-02.

## Tech Debt at Close

- **WR-01 (Phase 04):** Settlement filter `or_(dry_run==False, and_(dry_run==True, strategy_name.isnot(None)))` simplifies to True given the outer `strategy_name == :name` clause. Dead code, not broken.
- **WR-02 (Phase 04):** `open_trades` counts only `status=='dry_run'` while `total_trades` counts all rows. Brittle if non-dry-run strategy attribution lands later.
- **WR-03 (Phase 04):** `/api/strategies-summary` orphan ordering is non-deterministic.
- **WR-04 (Phase 04):** Analytics page lacks `popstate` listener — back/forward does not re-sync `selected` state. **Deferred to Phase 999.1** in ROADMAP backlog.
- **Concern (carried from Phase 03):** `connect_args timeout=5` is the only buffer against `SQLITE_BUSY` under analytics polling. If pressure increases, switch to WAL or move analytics to a read replica.
