<objective>
Close the two major gaps from `01-HUMAN-UAT.md` (Test 3 and Test 6) discovered during human verification of Phase 1 Plan 01.

**Gap 1 (Test 3):** Display sort uses date alone with no within-day tiebreaker. V8's stable sort preserves the source-JSON insertion order, so within a single day, trades render top→bottom in chronological order while across days they render in reverse-chronological order. The topmost row is therefore not the chronologically last trade. Fix by replacing the date-keyed sort with a single `Array.prototype.reverse()` of the chronological-walk insertion order — this yields strict newest-first across BOTH day boundaries AND within-day.

**Gap 2 (Test 6):** Per the user's Option-A-extended decision in `01-HUMAN-UAT.md` Test 6, zero-contract rows must be excluded from `wins`/`losses`/`matches_bet_on` tallies (so summary cards reflect actual edge captured, not hypothetical fires) AND must render as a third visual state (orange tint, no checkmark/cross icon) to distinguish "would have bet if capital permitted" from "did bet and won/lost".

**Design decision for Gap 2 (justified inline below):** Keep the `result: "win" | "loss"` discriminator unchanged (it carries the underlying match outcome — useful for future "would-have-won rate when capital-starved" analytics) and gate both the tally aggregation and the render path on `trade.contracts > 0` / `trade.contracts === 0`. Smaller diff than introducing a `result: "noop"` enum value, no `FireOutcome` schema change, and `contracts === 0` becomes the single source of truth for "actually bet vs not".

Purpose: Restore display invariants Phase 1 Plan 01 broke (or never enforced) — capital monotonicity walking the visible row list, summary tallies that match the user's mental model of "what did this strategy actually do", and a visual signal that capital-starvation events are categorically different from real losses.

**Gap 1 root cause:** `dashboard/app/backtest/backtest.ts:198-199`
```ts
const display_trades = [...trades].sort((a, b) =>
  b.date.localeCompare(a.date),
);
```
`runBacktest` walks `chronological` (line 138, sorted asc by `a.date.localeCompare(b.date)` only) and pushes each trade in walk order. V8's sort is stable, so within-day order in `chronological` is whatever order `file.matches` had it (source JSON order). The descending date sort on line 198 flips day groups but preserves within-day asc order. Result: top of list shows the chronologically *first* match of the most recent day, not the *last* match overall.

**Concrete UAT evidence (Aug 24, 2024 group as displayed):**
- 1004.59 ← Aston Villa 0–2 Arsenal (chronologically first on Aug 24)
- 1006.12 ← Manchester City 4–1 Ipswich Town
- 1007.65 ← Tottenham Hotspur 4–0 Everton (chronologically last on Aug 24)
- 1003.06 ← Aug 18 Chelsea 0–2 Manchester City
- 1001.53 ← Aug 17 Everton 0–3 Brighton

Walking top→bottom, capital reads 1004.59 → 1006.12 → 1007.65 → 1003.06 → 1001.53. The 1007.65 → 1003.06 step is correct (Aug 24 final → Aug 18); 1004.59 → 1006.12 → 1007.65 is wrong (within-day ascending instead of descending). Topmost row is NOT the final trade.

**Fix:** `[...trades].reverse()` instead of `.sort()`. Since `trades` is built by walking `chronological` in ascending order, reversing gives strict newest-first INCLUDING within-day. The topmost row becomes the chronologically last trade in the season. Capital walks monotonically (or flat for zero-contract) top→bottom.
