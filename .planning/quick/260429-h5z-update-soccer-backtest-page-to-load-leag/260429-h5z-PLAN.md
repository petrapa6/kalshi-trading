Verified shape of resources/epl_2024_25_season.json (the only file today, 380 matches, ~148 KB):
```ts
type SeasonFile = {
    matches: {
        id: string;            // e.g. "704279"
        date: string;          // "YYYY-MM-DD"
        home_team: string;     // canonical full name, e.g. "Manchester United"
        away_team: string;
        final_score: string;   // "h:a", e.g. "1:0"
        goals: {
            time: string;      // "MM|S" — base minute | stoppage minutes (both ints)
            score: string;     // "h:a" snapshot AFTER this goal
        }[];
    }[];
};
```

Filename convention: `{league}_{startYY|YYYY}_{endYY}_season.json`. Today: `epl_2024_25_season.json`. The plan parses with regex `^([a-z0-9]+)_(\d{4})_(\d{2})_season\.json$` so a 4-digit start year + 2-digit end year is required; anything that doesn't match is silently skipped (logged to console in dev).

Strategy logic (simplified port of src/predictions/backtest.py::simulate_match, dropped price/bankroll):
- Walk the goals chronologically (already chronological in the JSON).
- Maintain (home, away) score from each goal's `score` field (authoritative; do not recompute from team attribution because the JSON does not encode goal attribution).
- After each goal, evaluate the trigger: `minute >= min_minute` AND `abs(home - away) >= min_lead`. **Decision: use base-minute only** (the integer before `|`) for the `min_minute` comparison. Stoppage is ignored. This matches the Python backtest, which iterates `range(1, 91)` over base minutes.
- Fire on the first matching goal; subsequent goals in the same match are ignored.
- Win iff `final_score`'s leading side matches the side leading at fire time (compute both from `final_score` parsed as "h:a").
- Output per trade: `{ match_id, date, home_team, away_team, final_home, final_away, fired_at_minute, score_at_fire_home, score_at_fire_away, leading_side, result }`.
- Output summary: `{ matches_scanned, matches_bet_on, wins, losses, win_rate }`. No price-derived fields.

    Exports:

    ```ts
    export interface BacktestTrade {
        match_id: string;
        date: string;
        home_team: string;
        away_team: string;
        final_home: number;
        final_away: number;
        fired_at_minute: number;
        score_at_fire_home: number;
        score_at_fire_away: number;
        leading_side: "home" | "away";
        result: "win" | "loss";
    }
    export interface BacktestSummary {
        matches_scanned: number;
        matches_bet_on: number;
        wins: number;
        losses: number;
        win_rate: number; // 0..1, 4 decimals
    }
    export interface BacktestResult { summary: BacktestSummary; trades: BacktestTrade[] }
    export function parseGoalTime(time: string): { minute: number; stoppage: number };
    export function parseScore(score: string): { home: number; away: number };
    export function simulateMatch(match: Match, params: BacktestParams): BacktestTrade | null;
    export function runBacktest(file: SeasonFile, params: BacktestParams): BacktestResult;
    ```
    Behavior (mirror the relevant slice of `src/predictions/backtest.py::simulate_match`, no prices):
    - `parseGoalTime("45|2") -> { minute: 45, stoppage: 2 }`. Both must be non-negative ints; throw otherwise.
    - For the trigger comparison, **use `minute` only** (ignore `stoppage`) — documented in the goal-time decision above. Add a one-line comment in `simulateMatch` next to the comparison: `// base-minute only; stoppage ignored to mirror src/predictions/backtest.py`.
    - `simulateMatch` walks `match.goals` in order, parses each `score` as the AFTER-goal snapshot (authoritative), and on the first goal where `minute >= min_minute && abs(home - away) >= min_lead` returns a `BacktestTrade` (the "fired" event). Compute `result` by parsing `match.final_score`:
      ```
      finalHome > finalAway && leading === "home" => "win"
      finalAway > finalHome && leading === "away" => "win"
      otherwise (incl. final draw) => "loss"
      ```
    - `runBacktest` aggregates over `file.matches`, returning `{ matches_scanned: file.matches.length, matches_bet_on, wins, losses, win_rate: settled === 0 ? 0 : Math.round((wins/settled) * 1e4) / 1e4 }`.

    Why drop initial_balance and bet_percent (per constraints): they are dollar-denominated bet-sizing knobs whose only purpose was to drive the bankroll curve and dollar P&L — both of which require Kalshi prices we no longer have. Keeping them as dead UI controls would be misleading. Win rate is the only price-free, well-defined output.
